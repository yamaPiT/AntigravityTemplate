# ADAプロセス ドキュメント作成ガイドライン

## 1. 目的と適用範囲
本ガイドラインは、ADA (Agent-Driven Agile) プロセスにおいて、AIエージェントが開発ドキュメントを作成・更新するための軽量なルールブックである。
作成対象となるドキュメントは以下の3種類に限定する。
1. **ソフトウェア要求仕様書 (SRS)**
2. **アーキテクチャ設計書 (Architecture Design)**
3. **統合テスト仕様書 (Integration Test Specification)**

## 2. エージェントへの基本ルール（Agent Instructions）
* **役割の遵守:** 各ドキュメントを作成・レビューする際は、自身に割り当てられた役割（`requirements-engineer`, `architect`, `tester`等）のスキル定義（`SKILL.md`）に厳密に従うこと。
* **一貫性と追跡可能性:** 要求仕様、アーキテクチャ設計、統合テストは相互にリンクしている必要がある。設計書は要求仕様を満たし、テスト仕様書はそれらを検証できる内容にすること。
* **出力形式:** 原則としてMarkdown形式で出力し、可読性を高く保つこと。

---

## 3. ドキュメント別 標準目次構造

### 【A】 ソフトウェア要求仕様書 (SRS)
システムが「何をするか（What）」を定義する。
* **1. 導入:** 目的、適用範囲、用語定義
* **2. 全体記述:** システム環境、ユーザー特性、前提条件と制約事項
* **3. 機能要件:** ユースケースや機能群ごとの入力・処理・出力
* **4. 外部インターフェース要件:** UI、API、ハードウェア・通信インターフェース
* **5. 非機能要件:** 性能、信頼性、セキュリティ、保守性

### 【B】 アーキテクチャ設計書 (Architecture Design)
要求仕様（SRS）を満たすために、システムを「どう作るか（How）」の全体構造を定義する。
* **1. 導入:** 目的、対象範囲、参照する要求仕様書(SRS)
* **2. システムアーキテクチャ:** * 2.1 全体構成図（コンポーネントの配置と関係性）
  * 2.2 技術スタック（言語、フレームワーク、データベース等）
* **3. コンポーネント設計:**
  * 3.1 各コンポーネントの役割と責任
  * 3.2 コンポーネント間のインターフェースと通信方式
* **4. データアーキテクチャ:**
  * 4.1 データモデル（ER図に相当する構造）
  * 4.2 データの保存先とライフサイクル
* **5. 非機能設計:**
  * 5.1 セキュリティ方針（認証・認可の実装方針）
  * 5.2 エラーハンドリングとロギング方針

### 【C】 統合テスト仕様書 (Integration Test Specification)
コンポーネント間の連携や、システム全体が要求仕様（SRS）通りに動作するかを検証する計画と項目を定義する。
* **1. 導入:** テストの目的、対象範囲、テスト環境の構成
* **2. テスト方針:**
  * 2.1 統合の戦略（どの順番でコンポーネントを結合するか）
  * 2.2 テストデータの準備方針
* **3. テスト項目（テストケース）:**
  * （※各ケースには、識別子、目的、前提条件、入力手順、期待される結果を明記すること）
  * 3.1 正常系テスト（主要なユースケースの通しテスト）
  * 3.2 異常系テスト（エラー時の挙動、境界値テスト）
  * 3.3 インターフェーステスト（外部システムやAPIとの結合テスト）
* **4. 合否判定基準:** テストを通過したとみなすための明確な基準（カバレッジ、エラー発生率等）

## 4. レビュー基準 (Review Criteria)
各種レビューア（`reviewer`）は、以下の観点でドキュメントを評価する。
1. **完全性:** 各ドキュメントの標準目次構造が網羅されているか。
2. **一貫性:** 要求(SRS) → 設計(Arch) → テスト(Test) の間で内容に矛盾や飛躍がないか。
3. **検証可能性:** 特に要求仕様と設計において、テストケースに落とし込める明確な条件が定義されているか。
